# Project Scaffold & Framework Rules System — Design Spec

Date: 2026-04-30

## Problem

LLM-generated projects lack engineering discipline:
- No standard project structure — layout varies arbitrarily between runs
- Framework rules from PRD (e.g. "PyQt6 ONLY") are lost during plan→decompose→execute pipeline
- No mechanism to inject known framework pitfalls (e.g. PyQt6 scoped enum syntax)
- Validation only checks imports and SQL columns, not structural or framework rule violations

Root cause: the workflow has three injection points (task, schema, contract) but none carries framework rules or engineering standards.

## Solution Overview

Add a **Scaffold phase** and **two-layer rules system** to the DevWorkflow pipeline:

```
PRD → Scaffold → Plan → Decompose → Contracts → Execute(含守卫) → Report → Smoke Test
```

Three components:
1. **Two-layer AGENT.md** — universal engineering standards (fixed) + project-specific rules (extracted from PRD)
2. **Scaffold phase** — creates project skeleton and project AGENT.md without LLM calls
3. **Real-time guard** — validates framework rules and structural integrity after each task

## Component 1: Two-Layer AGENT.md

### Layer 1: simple-agent AGENT.md (fixed, ships with simple-agent)

Located at `src/simple_agent/agent_md_template.md`. Defines universal engineering discipline applicable to ALL generated projects:

```markdown
# simple-agent Engineering Standards

## Required Project Elements
Every generated project MUST have:
- Entry point (main.py / app.py / index.ts / etc.)
- Dependency declaration (requirements.txt / package.json / pyproject.toml)
- AGENT.md (project-specific rules)
- tests/ directory
- .gitignore

## Quality Gates
- All Python files must pass import check
- Database projects: schema file is single source of truth
- Smoke test must pass (app starts without crash)
- AGENT.md must not be modified or deleted during execution

## Code Standards
- No placeholder code (TODO, FIXME, pass-only functions)
- No hardcoded secrets or credentials
- Error messages must be user-friendly
```

This file is bundled with simple-agent. It does NOT contain technology-specific rules.

### Layer 2: Project AGENT.md (dynamic, generated per project)

Located at `{project_root}/AGENT.md`. Generated by the scaffold phase from:
- PRD Architecture section → framework and tech stack
- PRD Conventions/Rules sections → project-specific constraints
- Framework rules knowledge base → known pitfalls for detected frameworks

Example output:
```markdown
# Library Management System — Project Rules

## Framework
- PyQt6 ONLY (do NOT use PyQt5, do NOT mix versions)
- PyQt6 scoped enum syntax: QTabWidget.TabPosition.North (NOT QTabWidget.North)
- All imports: from PyQt6.QtWidgets import ...

## Architecture
- Layered: UI → Service → DatabaseManager
- MainWindow creates all services, passes to views via constructor
- No raw SQL in UI code

## Testing
- Unit tests for each service (tests/test_*_service.py)
- Smoke test: app must start in offscreen mode
```

## Component 2: Scaffold Phase

### Position in Pipeline

Inserted BEFORE Plan phase:

```
PRD → Scaffold → Plan → Decompose → Contracts → Execute → Report → Smoke Test
```

Rationale: Plan and all subsequent phases operate within the scaffolded directory.

### What Scaffold Does (NO LLM calls — pure structured operations)

1. **Parse PRD** — extract structured sections via regex/heading matching:
   - `Architecture` section → detect tech stack keywords
   - `Data Model` section → note table names for schema file
   - `Conventions` / `Rules` sections → extract constraint text

2. **Create project skeleton**:
   ```
   {project}/
   ├── AGENT.md              ← generated project rules
   ├── main.py               ← empty entry point
   ├── requirements.txt      ← dependencies inferred from Architecture
   ├── database_init.sql     ← empty (if Data Model exists)
   ├── src/                  ← empty directory
   ├── tests/                ← empty directory
   └── .gitignore            ← generic template
   ```

3. **Generate project AGENT.md**:
   - Extract rules from PRD Conventions/Rules sections
   - Detect frameworks from Architecture section
   - Append matching framework rules from knowledge base

### Framework Rules Knowledge Base

```
src/simple_agent/framework_rules/
├── pyqt6.md      # scoped enums, signal syntax, thread safety
├── flask.md      # session security, SQL injection, blueprint patterns
├── fastapi.md    # async pitfalls, dependency injection, Pydantic v2
├── react.md      # hooks rules, key prop, state immutability
└── generic.md    # error handling, logging, security
```

Detection logic: scan PRD Architecture section for keywords:
- `PyQt6` / `PyQt5` / `PySide` → load pyqt6.md
- `Flask` → load flask.md
- `FastAPI` → load fastapi.md
- `React` → load react.md
- Always append generic.md

## Component 3: Prompt Injection & Real-time Guard

### Prompt Injection

Current task augmentation (dev_workflow.py):
```python
augmented_task = f"{task.description}{schema_block}{contract_block}"
```

New task augmentation:
```python
rules_block = build_rules_block(project_agent_md, simple_agent_agent_md)
augmented_task = f"{task.description}{rules_block}{schema_block}{contract_block}"
```

Where `rules_block`:
```
---
## Project Rules (from AGENT.md — STRICTLY follow)
{project_agent_md_content}
---
## Engineering Standards (non-negotiable)
{simple_agent_md_content}
---
```

Injection points (all places where tasks are executed):
- `execute()` — normal execution
- `resume()` — resume after pause
- `retry_failed()` — retry failed tasks

### Real-time Guard

Enhanced `_validate_task_output` adds checks after each task:

| Check | Description | Failure behavior |
|-------|-------------|-----------------|
| AGENT.md integrity | Key sections not deleted/emptied | Mark task failed |
| Forbidden import detection | Scan .py files for imports violating AGENT.md framework rules | Mark task failed + specific error |
| Directory guard | tests/ directory must exist | Mark task failed |
| Empty file check | No placeholder-only function bodies | Warning only |

### Forbidden Import Detection

Parse project AGENT.md Framework section for rules like:
```
PyQt6 ONLY (do NOT use PyQt5)
```

Extract: `allowed = ["PyQt6"], forbidden = ["PyQt5", "PySide2"]`

Scan all .py files for `import {forbidden}` or `from {forbidden}` via regex.

## CLI Changes

### New Argument

```bash
--skip-scaffold    # Skip scaffold phase (for existing projects)
```

### Full Command Examples

```bash
# Full workflow with scaffold
python build_with_workflow.py demo/prd.md -o demo/project

# Skip scaffold for existing project
python build_with_workflow.py demo/prd.md -o demo/project --skip-scaffold

# Existing flags unchanged
python build_with_workflow.py demo/prd.md -o demo/project --fix
python build_with_workflow.py demo/prd.md -o demo/project --retry
```

Add section at end of report:

```markdown
## Scaffold & Rules

- Rules file: AGENT.md (12 rules, 3 pitfall warnings)
- Guard checks: 4 passed, 0 failed
```

## Out of Scope

- Auto-detection of framework from existing codebase (only PRD-based detection)
- Multi-language project scaffolding (Python-first, extensible later)
- IDE integration or editor templates
- Version migration rules (e.g. PyQt5 → PyQt6 upgrade guidance)
